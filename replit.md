# RAYO Platform
RAYO (anteriormente RAIO; renomeado em Maio/2026) é uma plataforma digital para fortalecer famílias através de conteúdo transformador, comunidade engajada e recursos práticos em cinco contextos de vida: Solteiro, Namoro, Noivos, Casados, Pais.

## Run & Operate
- `npm run dev` (Express + Vite middleware) · `npm run build` (frontend prod).
- **Env vars principais**: `resend_api_key`, `RESEND_FROM_EMAIL`, `APP_URL`, `PUBLIC_SITE_URL` (default `https://rayo.app.br`), `ADMIN_EMAILS`, `CONTACT_EMAIL` (alias legado `CONTATO_TO_EMAIL`).
- **Bunny Stream**: `BUNNY_STREAM_LIBRARY_ID/API_KEY/CDN_HOSTNAME/WEBHOOK_SECRET` — ver `docs/contracts/bunny-stream.md`.
- **OAuth Google/Facebook** (só prod `https://rayo.app.br`): `GOOGLE_CLIENT_ID/SECRET/REDIRECT_URI`, `FACEBOOK_CLIENT_ID/SECRET/REDIRECT_URI`. Sem `*_REDIRECT_URI` exato → botão "Em breve".
- **Stripe (Trilhas pagas — Task #130)**: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET` (gerado/managed pelo `stripe-replit-sync` no boot, opcional). Endpoint webhook: `POST /api/stripe/webhook` (express.raw, registrado ANTES do express.json em `server/index.ts`). Frontend usa Stripe Checkout hosted (sem chave pública necessária no client).

## Stack
- **Frontend**: React 18, TypeScript, Tailwind v4, Vite. Fonte única **Outfit** (200–700).
- **Design System v2.0**: `src/styles/` — `home-rayo.css` (`.rh-*`), `nav-rayo.css` (`.rn-*`), `playlists-rayo.css` (`.rh-pl-*`), base `app-rayo.css` (`.ra-*`). Tokens `--rayo-*` em `globals.css` (legacy `--raio-*` removido em #60).
- **Backend**: Express + TS + tsx. **DB**: PostgreSQL. **Auth**: sessão com cookies seguros. **Email**: Resend.

## Architecture decisions
- **Thin Client, Fat Server**: lógica/auth/dados/segurança no Express.
- **Idempotent Migrations/Seeds**: migrações de boot são idempotentes.
- **RBAC**: hierarquia numérica (`client` < `producer` < `moderator` < `admin`).
- **Soft Deletes**: conteúdo é `hidden`, não deletado.
- **SSE para DM**: real-time messaging via Server-Sent Events.
- **Dynamic Home Feed**: `recommendedSectionOrder` por segmentos; "Hoje no RAYO" usa rotação determinística diária.
- **Static Assets em Object Storage**: sentinels `objstore://<key>` resolvidos via signed URL em leitura. Imagens passam por `sharp`.
- **Vídeo via Bunny Stream**: sentinel `bunny://<libraryId>/<guid>` — ver `docs/contracts/bunny-stream.md`.

## Product
- **Auth**: registro, login, logout, reset, perfil, verificação de e-mail.
- **CMS**: 6 kinds (audio, video, reels, serie, curso, livro) + `artigo` (blog — ver `docs/contracts/marketing-seo.md`).
- **Gamification**: XP, badges, missões, streaks. **Personalized Dashboard** agrega tudo + comunidade.
- **Community Forums**: subreddit-style — ver `docs/contracts/comunidade.md`.
- **Direct Messaging**: per-side state + attachments + swipe — ver `docs/contracts/dm.md`.
- **Notificações**: tabela `notifications` (kind/title/body/link/payload), `NotificationBell` no header, SSE (`notification:new`/`unread`), endpoints `/api/notifications`.
- **LGPD**: export e deleção de conta em `/api/lgpd`.
- **Admin & Moderation**: gestão de usuários, moderação, métricas.
- **Turmas (mini-Skool)**: ver `docs/contracts/turmas.md`.
- **Trilhas pagas (Stripe — Task #130)**: 5 trilhas curadas por momento de vida (Solteiro/Namoro/Noivos/Casados/Pais), cada uma agrupando N turmas. Assinatura recorrente Stripe (mensal ou anual com ~20% off). Tabelas: `trails`, `trail_courses` (M:N), `subscriptions` (fonte da verdade — atualizada por webhook), `users.stripe_customer_id`. Catálogo público em `/trilhas`, detalhe `/trilhas/:slug`, sucesso pós-checkout `/trilhas/sucesso?slug=…`. Comunidade pública continua grátis; turmas vinculadas a trilha paga (e a comunidade `class_id` delas) são gated. Admin em "Trilhas (Stripe)" no AdminShell. "Minhas assinaturas" no Perfil chama `/api/billing/portal` (Stripe Customer Portal). Contrato completo em `docs/contracts/billing.md`.

## Gotchas universais
- **Route Order**: rotas com prefixo fixo antes das dinâmicas no Express.
- **Email Enumeration Prevention**: forgot password sempre retorna sucesso.
- **Object-Level Authorization**: producers só editam conteúdo próprio; `moderator+` override.
- **Content Card Mapping**: `badge_text`, `meta_text`, `progress`, `gradient` têm significado contextual.
- **No Fake Discounts**: preço de curso vem de `course.price`; promoções só do backend.
- **Rate Limiter (Task #51)**: `rateLimiter(max, windowMs, opts)` em `server/middleware/security.ts` — bucket por instância, `opts.keyByUser` (cai pra IP quando anônimo) e `opts.skip(req)`. `optionalAuth` hidrata `req.user` antes de cada limiter autenticado. `app.set("trust proxy", 1)`. `/api/auth` POSTs sensíveis 20/15min por IP; resto 60/15min keyByUser (`GET /me` skipado). LGPD em `/api/lgpd`. Demais autenticadas 120–600/15min keyByUser.
- **Idempotent Daily Completions**: `POST /api/home/today/complete` usa `ON CONFLICT DO NOTHING`.
- **Re-tap aba ativa = scroll-top (Task #115)**: `Navigation`/`DesktopSidebar` checam `isActive` e disparam `dispatchScrollTop(tabId)` em vez de `onTabChange` quando o usuário re-clica a aba já ativa. Listener global em `App.tsx` rola `window`. Páginas com side-effects extras escutam via `onScrollTop()` (`src/lib/scrollTop.ts`): `ComunidadePage` reseta `activeCommunitySlug`, `PerfilPage` fecha o overlay de perfil público, `ConversasPage` fecha a conversa aberta. Não é "double-tap" temporal — é re-tap de aba já ativa, simples e previsível.
- **Cards de perfil clicáveis (Task #115)**: Posts/Comentários/Salvos no `UserProfilePage` viraram `role="button"` que disparam `rayo:open-post` (mesmo CustomEvent da busca) + sessionStorage `rayo-pending-post`. Comentários incluem `highlight_comment_id` (e key extra `rayo-pending-post-comment`); o `CommentsPanel` em `ComunidadePage` rola até `[data-comment-id="<id>"]` e aplica `.rayo-comment-highlight` (animação 2,2s definida em `globals.css`). Botões internos (`c/<slug>`) usam `e.stopPropagation()` pra não disparar o click do card. **Task #163**: storage/eventos renomeados de `raio:*`/`raio-*` → `rayo:*`/`rayo-*` no rebrand. Migração one-shot em `src/lib/storageMigration.ts` (flag `rayo_storage_migrated_v1`) copia chaves legadas no boot do `App.tsx`. sessionStorage usa fallback de leitura (`rayo ?? raio`) por uma janela de transição; CustomEvents foram trocados atomicamente (single SPA, sem dual-emission).
- **Reações multi-emoji em posts/comments (Task #122)**: set fechado `["❤️","😂","🙏","💡","🔥","👏"]` espelhado em `ALLOWED_REACTION_EMOJIS` (server/features/community/service.ts) e `REACTION_EMOJIS` (src/components/EmojiReactionPicker.tsx). Tabelas `post_reactions`/`comment_reactions` (UNIQUE user+target+emoji) com backfill ❤️ a partir do legado `post_likes`/`comment_likes`. Endpoints novos `POST /api/community/{posts,comments}/:id/reactions` (toggle/swap/remove) devolvem `{reactions:[{emoji,count}], user_reaction}`. `like_count` continua somando o TOTAL de reações (preserva trending/karma); `togglePostLike`/`toggleCommentLike` viraram aliases pra `❤️`. `getPostDetail`, `hydratePostsRows` e `dashboard.recentPosts` hidratam `reactions[]`+`user_reaction` em batch via helpers `aggregateReactions`/`userReactionsFor`. Frontend: `EmojiReactionPicker` (variants `full`/`compact`) é all-in-one — botão+popover+chips agregados, faz a request e devolve estado via `onChange`. PostCard mantém estado local de reações (não dispara `loadPosts` global). CommentsPanel passa `onCommentReactionsChange(commentId, next)` pro pai. Discussão dedicada compartilhável: `/c/<slug>/p/<id>` em `App.tsx` parqueia `rayo-pending-post`+`rayo-pending-discussion-slug`, ComunidadePage abre o CommentsPanel; `publicMeta.ts.resolveCommunityPost` devolve OG/JSON-LD `DiscussionForumPosting`. `FavoriteIcon` virou Bookmark (Heart só restou no `EmojiReactionPicker` via emoji ❤️) pra acabar com o "dois corações" no card.
- **Stripe Billing — gating + webhook (Task #130)**: Webhook `POST /api/stripe/webhook` é registrado com `express.raw({type:'application/json'})` ANTES do `express.json()` global (linhas 73-92 de `server/index.ts`) — sem isso a verificação HMAC quebra. Init no `start()`: `migrateBilling()` → `runMigrations()` (stripe-replit-sync) → `findOrCreateManagedWebhook()` → `syncBackfill()`. **Fonte da verdade = tabela `subscriptions`** (status active|trialing|past_due valem como acesso); processamos via `handleEvent` (idempotente por `stripe_subscription_id`) DEPOIS de delegar pro `processWebhook` do sync. Metadata Stripe `rayo_user_id` + `rayo_trail_id` é setada no Checkout Session — o webhook usa pra upsert. Stripe não permite editar valor de price → `service.ts` cria price novo + desativa antigo. Cache 60s `course_id → trail_id` em `service.ts` (`invalidateTrailLookupCache` chamado em writes/webhooks). Gating em `server/middleware/requireTrailAccess.ts` exposto via helper `checkCourseAccess(req, courseId)`. `server/features/academia/routes.ts` usa o helper local `ensureTrailAccess` que devolve 402 `TRAIL_PAYMENT_REQUIRED` (com `trail_id`) em `POST /:id/enroll`, `PATCH /lessons/:id/progress` e `GET /:id/progress`. Comunidade fica gated transitivamente: posts com `class_id` exigem matrícula via `isCourseMember` e enrollment já passou pelo gate. Frontend captura o 402 mostrando `<TrailPaywall trailId={…}/>`. Preços em CENTAVOS BRL (`monthly_price_cents`, `yearly_price_cents`). Rotas `/trilhas`, `/trilhas/:slug`, `/trilhas/sucesso` rodam na **PublicShell** (sem AuthProvider) — `TrilhaDetailPage` chama `/api/auth/me` direto pra detectar sessão sem depender do contexto.
- **DM viewport sizing (Task #144)**: A página de conversas precisa de altura travada na viewport — sem isso, `.ra-page` (min-height:100vh) deixa o shell crescer com o conteúdo e o composer/header param de ficar fixos (a janela inteira passa a rolar). Regra: `.rayo-dm-shell.ra-page { height:100dvh; min-height:0; overflow:hidden }`. Desktop desconta 80px da TopNavbar via `calc(100dvh - 80px)`. Toda coluna flex interna precisa de `min-h-0` (esquerda: lista; direita: conversa+overflow:hidden) pra a `<ScrollArea>` interna conseguir encolher e rolar por dentro. `ConversasPage` adiciona `body.rayo-dm-page` (sempre que monta) que zera `padding-bottom` do `<main>` (`.desktop-layout*` e `.navbar-bottom-spacing`) — sem isso a soma main_padding + shell_height excede a viewport e a página rola 80-112px. `body.rayo-dm-conversation-open` (só com convo aberta) esconde a bottom nav mobile. Estado lista-only no mobile/tablet (≤1023px) reserva 72px+safe-area dentro do shell pra última conversa não cair atrás da bottom nav. Regra geral pra qualquer página tipo "app shell": se a UI precisa de header+footer fixos com área central rolável, o container raiz não pode usar min-height — tem que usar height travada e ancestrais flex precisam de min-h-0.
- **DM polish — bubble fit-content + AudioBubble (Task #145)**: Balão de texto **NUNCA** pode ter `max-width: %` direto em `.ra-chat-bubble` — em flex-col aninhado isso colapsa pra `min-content` e quebra texto curto letra a letra na vertical. Regra: `.ra-chat-bubble { width: fit-content; max-width: 100%; min-width: 0; overflow-wrap: anywhere }` e o wrapper flex-col carrega o cap real (`w-fit max-w-[min(80%,560px)]`). `<audio controls>` nativo foi banido das DMs — usar `<AudioBubble>` (`src/components/AudioBubble.tsx`): botão play circular + barra clicável + tempo, hooks `onPlay/onTimeUpdate` chamam `sendListeningPing`. Variantes `user|assistant|compact` (compact pro preview do composer). Status leitura: ícone-only colorido (`CheckCheck` terra-500 = lido, `Check` neutro = enviado), texto "Lido"/"Enviado" removido (aria-label preserva acessibilidade). Pílula de data com divisores horizontais (`::before`/`::after` flex:1 height:1px). `GROUP_WINDOW_MS` em `messageGrouping.ts` subiu de 2min→5min (menos repetição de "22:29"). Botão "Nova" trocou ícone `UserPlus` → `MessageSquarePlus`. Preview da lista de conversas mostra duração: `🎤 Áudio (0:07)` — exige `last_message_meta` no SELECT da query `listConversations` (não esquecer no LATERAL JOIN). Imagens DM têm `onError` que troca pelo placeholder `.ra-chat-attachment-fallback`. Botões "(em breve)" no `TopNavbar` (Criar conteúdo, Favoritos) e `DesktopSidebar` (Configurações) só aparecem pra `userHasRole(user, "producer")` — usuário final via card desabilitado sem contexto era ruído.
- **Catálogo conectado a Trilhas (Task #151)**: `GET /api/courses` faz LEFT JOIN `trail_courses + trails` e devolve `trail_id/trail_slug/trail_title` por curso (`server/features/academia/service.ts:listCourses`). No frontend (`AcademiaPage.tsx::CourseCard`), curso com `trail_slug` redireciona pra `/trilhas/:slug` (checkout Stripe real) em vez de abrir TurmaShell→JoinInterestModal/waitlist. Card mostra badge terra-500 "Trilha: X", esconde preço avulso e troca CTA pra "Ver trilha". `TurmaLandingPage` standalone (fora do TurmaShell) renderiza `<TrailPaywall>` quando `trail_id && !has_trail_access && !is_member`; senão CTA virou "Avise-me quando abrir" (copy honesta — não promete vaga). Em `MarketplaceView`, `displayedPopular`/`displayedTopRated` derivam de `segmentFilteredCourses` quando `selectedSegment !== 'all'` (antes ignoravam o filtro de segmento — bug de auditoria). `visibleFormatKinds` esconde formatos com `count=0` (Áudios/Reels seedados como ícones-fantasma); "Cursos" sempre visível porque é link de scroll. Rating/students do CourseCard só renderizam quando `> 0` (migração idempotente em `schema.ts` zera rating/students dos 6 cursos seed legados via match por `(title, rating)` — rating é estável; students cresce com matrículas reais e portanto não serve como discriminador). Preserva edits manuais no admin.
- **E2E Playwright (Task #160)**: suíte em `tests/e2e/` rodando contra o dev server real + DB de dev. Rodar com `npx playwright test`; ver `tests/e2e/README.md` pra detalhes. Especificidades do container Replit: (1) `chromium.launch()` morre por causa do `--inspector-pipe` — `global-setup.ts` lança o chromium do Replit (`$REPLIT_PLAYWRIGHT_CHROMIUM_EXECUTABLE`) com `--remote-debugging-port=0` e o fixture em `tests/e2e/fixtures.ts` conecta via `chromium.connectOverCDP`. Specs DEVEM importar `test`/`expect` de `./fixtures` (não de `@playwright/test`). (2) `waitForLoadState("networkidle")` nunca resolve (SSE sempre aberto) — espere por elemento âncora. (3) `registerUser` em `helpers/api.ts` insere linha verified em `email_verification_codes` antes do POST pra pular Resend. (4) Rate limiter `/api/auth` (20/15min por IP) é in-memory — restart `Start application` se bater limite. (5) Cleanup via `deleteUsersById`/`deleteAllTestUsersByEmailPrefix` (prefixo `test-`@rayo.test).
- **CommentsPanel via Portal (Task #115)**: `CommentsPanel` agora é renderizado via `createPortal(jsx, document.body)`. Sem isso, o `transform: translateY(...)` aplicado por `PullToRefresh` no wrapper de conteúdo cria um containing block que **quebra `position: fixed` em descendentes** — o painel aparecia "fora da viewport" no mobile. O painel também faz body-scroll-lock e fecha com Esc/click no backdrop. Regra geral: qualquer overlay/modal montado dentro de `PullToRefresh` precisa ir pro portal (ou então usar shadcn `Sheet`/`Dialog` que já fazem isso).
